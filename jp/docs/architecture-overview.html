FeelFuseSDK Architecture Overview

Project Summary

Project Name: FeelFuseSDK-dev Developer: Commissure Unity Version: 2022.3.62f1 Target Platforms: VR/AR (Meta Quest, OpenXR Compatible) Primary Purpose: Haptic feedback system for Unity VR/AR applications

Architecture Overview

1. Layered Architecture

The SDK employs a clear 4-layer architecture with well-defined responsibilities:

  • Infrastructure Layer: Unity Engine, Meta XR SDK, OpenXR
  • Platform Abstraction Layer: Meta/XRIT compatibility modules
  • Core Layer: Haptic system core functionality
  • Application Layer: Test scenes, prefabs, and samples

2. Modular Design

The system is divided into multiple independent assemblies, each with specific responsibilities:

Foundation Layer:

  • Base: Common utilities, registries, and shared interfaces (IStatus, IRegisterer, etc.)

Core Modules:

  • HapticsSystem.Core: Core haptic system interfaces and management
  • ModelTracking.Core: Body part modeling, tracking, and BodyPartTarget system

Device Layer:

  • HapticsSystem.Device: Device abstraction foundation
  • HapticsSystem.Device.V0/V1: Version-managed device implementations

Platform Compatibility:

  • ModelTracking.Hands_Meta/Hands_XRIT: Hand tracking system implementations
  • HapticsSystem.Interaction_Meta/Interaction_XRIT: Interaction system implementations

3. Extensible Design Patterns

Interface-Centric Design: All major components are interface-based, enabling easy implementation swapping and testing.

Generic Type System: IHapticManager<TValue>, IHapticInput<TValue>, etc., provide type safety while maintaining flexibility for different value types.

Registrar Pattern: IHapticSourceRegisterer enables dynamic component registration and deregistration.

Key Components

Base Module

Foundation layer providing common infrastructure:

  • Registries: Centralized component registration system
  • Common Interfaces: Shared interfaces like IStatus, IRegisterer, IDebuggable
  • Utility Classes: Helper functions and common utilities used across all modules
  • This module serves as the dependency foundation for all other modules

HapticManager

Central system coordinator responsible for:

  • Device registration and management
  • Input/output component coordination
  • System lifecycle management

BodyPartTarget System

Hierarchical body part identification system implemented in ModelTracking.Core:

All → Upper/Lower → Arm/Hand → Finger → Thumb/Index/etc → FirstPhalanx/etc

The BodyPartTarget system provides precise body part identification for haptic feedback targeting.

Device Abstraction

Three-tier device support:

  1. Mock: Testing and debugging
  2. V0: Legacy implementation
  3. V1: Current full implementation with OSC communication

Coordinate Transformation System

Recent enhancements for accurate haptic positioning:

  • ConvertToLocalForces: Transforms haptic force data from world coordinates to local device coordinates
  • Purpose: Ensures haptic feedback is correctly oriented relative to the device regardless of world space orientation
  • Implementation: Utilized in HapticForceOutputV0 with optimized cached list processing
  • Use Case: Critical for maintaining consistent haptic feedback during device movement and rotation

Platform Strategy

Conditional Compilation

Supports both Meta XR and Unity XR while eliminating unnecessary dependencies:

  • Meta_XR_Hands, Meta_XR_Interaction: Meta Quest optimization
  • Unity_XR_Hands: Cross-platform support

Deployment Options

  • Meta Quest: Native optimization for high performance
  • OpenXR Devices: Standardized API for broad compatibility
  • PC Development Environment: Mock devices for development efficiency

Development and Testing Support

Test Infrastructure:

  • Multiple test scenes (BlankScene, SampleScene, etc.)
  • UnityEventChecker for behavior verification
  • HapticSources prefabs for functionality testing

Debug Features:

  • LogFilter for detailed log control
  • Mock devices for behavior simulation
  • Real-time operation verification in editor

Design Benefits

  1. Maintainability: Modular design enables easy individual component updates
  2. Extensibility: Interface-based design simplifies new feature addition
  3. Compatibility: Multi-platform support enables broad deployment
  4. Quality Assurance: Rich test environment ensures stability
  5. Development Efficiency: Mock system enables development without hardware

Key Design Decisions

Interface Segregation

Interfaces are carefully segregated by responsibility:

  • Device management vs. data flow management
  • Basic vs. generic type-safe operations
  • Input vs. output concerns

Generic Type Safety

The <TValue> parameter enables type-safe haptic data processing while maintaining flexibility for different data types (audio, force, etc.).

Platform Abstraction

Two-path approach:

  • Meta-specific implementations for optimal Quest performance
  • Unity XR implementations for cross-platform compatibility

Version Management

Explicit versioning (V0, V1) allows:

  • Backward compatibility maintenance
  • Gradual migration paths
  • Feature evolution tracking

Technology Stack

Core Technologies:

  • Unity 2022.3.62f1
  • Meta XR SDK 78.0.0
  • OpenXR 1.14.3
  • Unity XR Management 4.5.2

Development Tools:

  • Unity Test Framework
  • Unity Memory Profiler
  • Assembly Definition Files (.asmdef)
  • Conditional compilation symbols

Future Extensibility

The architecture supports future expansion through:

  • Additional platform adapters (new XR platforms)
  • New device versions (V2, V3, etc.)
  • Extended body part models (full body tracking)
  • New haptic value types (thermal, electrical, etc.)

This structure makes FeelFuseSDK a practical and scalable implementation of haptic feedback technology for Unity VR/AR applications.
